# Tiny Platform SB4 / SB3 Branch Strategy

最后更新：2026-04-18

适用仓库：

- `/Users/bliu/code/tiny-platform`

## 1. 当前分支定位

- `sb4`
  - 当前唯一业务开发主干
  - 当前承载 Spring Boot 4 / Camunda 7 fork 兼容后的正常业务迭代、线上修复与治理变更
- `sb3`
  - 当前维护分支
  - 仅承接从 `sb4` 选择性回补的稳定业务修复与必要维护补丁
- `main`
  - 历史基线或默认入口分支
  - 在正式切换默认分支前可能仍保留仓库默认入口角色，但**不再作为日常开发、提测、修复、治理变更的默认目标分支**

一句话收敛：

- 现在：`sb4` 是主干，`sb3` 是维护线，`main` 只是历史入口
- 不是：`main` 开发后自动同步到 `sb4`

## 2. 改动分类规则

### 2.1 主干业务改动

定义：

- 面向产品功能、权限、菜单、租户、审批、页面、接口、数据库演进等正常业务改动

规则：

- 默认从 `sb4` 拉分支开发并合入 `sb4`
- 需要维护线覆盖时，再从 `sb4` 选择性回补到 `sb3`
- 不再默认要求“先合 `main` 再同步”

### 2.2 `sb4-only` 兼容改动

定义：

- 仅为 Spring Boot 4 / Camunda 7 fork / Jakarta / Jackson 3 / 测试基础设施 / CI 兼容而存在的改动
- 在 `sb3` 的 Spring Boot 3 语义下不需要、也不应落地的改动

规则：

- 只进入 `sb4`
- 不回补 `sb3`
- PR 中应明确标注为 `sb4-only`

典型例子：

- Camunda fork 版本切换
- Boot 4 starter / 自动配置兼容修复
- Jakarta / Jackson 3 迁移
- 仅服务于 `sb4` 的 CI、runbook、收口文档

### 2.3 `sb4` 到 `sb3` 的维护回补

定义：

- 改动首先在 `sb4` 收敛并验证通过，但 `sb3` 维护线也需要该稳定修复

规则：

- 必须从 `sb3` 拉出**专用 backport 分支**
- 必须使用 `cherry-pick -x`、定向 backport 提交或等价可追踪方式
- 必须重新在 `sb3` 上执行与改动范围匹配的验证
- PR 中必须记录来源 `sb4` 提交 SHA 与回补原因
- **禁止** 直接创建整线 `sb4 -> sb3` 周期性同步 PR 代替选择性回补

## 3. 日常工作流

### 3.1 主干业务改动标准路径

1. 从 `sb4` 拉业务分支开发
2. PR 合入 `sb4`
3. 若确认 `sb3` 也需要该修复，再单独创建 backport 分支
4. 将所需提交从 `sb4` `cherry-pick -x` 到该 backport 分支
5. 在 `sb3` 目标语义下重新验证并发起 `sb3` PR

### 3.2 `sb4-only` 改动路径

1. 从 `sb4` 拉兼容分支开发
2. PR 合入 `sb4`
3. 在 PR 中明确写明“不回补 `sb3`”

### 3.3 `sb3` 维护回补路径

1. 从 `sb3` 拉出 backport 分支
2. 从 `sb4` 挑选稳定提交 `cherry-pick -x`
3. 重新运行 `sb3` 目标分支所需验证
4. 发起 `sb3` PR，并记录来源 SHA / 风险 / 未回补项

## 4. 硬性约束

- `sb4` 与 `sb3` 均应开启分支保护，禁止直接 push
- `sb4` 是唯一默认业务主干；不要把日常业务需求首发到 `main`
- `sb3` 只接稳定维护修复，不能充当双写开发主线
- `sb4` 到 `sb3` 的回补必须是**选择性**的，不能用整线同步替代
- 只有明确的兼容性工作，才能标记为 `sb4-only`
- 若某修复要进入 `sb3`，则必须能够说明“为什么 `sb3` 也需要它”

## 5. 推荐标签口径

建议在仓库中维护以下标签：

- `backport-to-sb3`
  - 已在 `sb4` 收敛，准备或正在回补到 `sb3`
- `sb4-only`
  - 仅适用于 Spring Boot 4 / Camunda fork / Jakarta / Jackson 3 兼容线

说明：

- 正常的 `sb4` 主干业务 PR 可以不额外打同步标签
- 只有当某修复需要回补 `sb3` 时，才使用 `backport-to-sb3`

## 6. 提交前缀建议

目标：

- 让提交标题本身就能表达“这是 `sb4` 主干业务改动”“这是 `sb4-only` 兼容改动”还是“这是 `sb3` 回补”
- 降低把维护回补误读成全新功能提交的概率

推荐口径：

- `sb4` 主干业务改动：
  - `feat: ...`
  - `fix: ...`
  - `refactor: ...`
  - `docs: ...`
- `sb4-only` 兼容改动：
  - `feat(sb4): ...`
  - `fix(sb4): ...`
  - `chore(sb4): ...`
  - `ci(sb4): ...`
  - `docs(sb4): ...`
- `sb3` 维护回补：
  - `fix(sb3): backport ...`
  - `chore(sb3): backport ...`
  - 或 `cherry-pick -x` 后在 PR 中说明来源 SHA
- 分支治理 / 共享规则更新：
  - `docs(governance): ...`
  - `chore(governance): ...`

避免项：

- 不要把 `sb3` 回补提交写成全新主干特性
- 不要把 `sb4-only` 兼容提交写成无分支语义的普通业务提交

一句话判断：

- 如果这是现在和以后默认开发都要依赖的业务改动，就进 `sb4`
- 如果这是 `sb3` 维护线也必须拥有的稳定修复，就从 `sb4` 定向回补到 `sb3`
- 如果这是只服务 `sb4` 技术栈的兼容改动，就显式加 `(sb4)`

## 7. PR 描述要求

所有涉及 `sb4` / `sb3` 关系的 PR，建议至少写清：

- 本次改动属于“`sb4` 主干业务”“`sb4-only` 兼容”还是“`sb3` 回补”
- 来源分支与目标分支
- 是否需要回补到另一条长期分支
- 若是 `sb3` 回补，来源 `sb4` 提交 SHA 是什么
- 若暂不回补，谁负责判断和补齐

## 8. 自动化与工作流边界

仓库当前治理口径：

- `.github/workflows/open-backport-to-sb3-pr.yml`
- `docs/TINY_PLATFORM_SB4_BRANCH_ENABLEMENT_TASK_CARD.md`

默认行为：

- **不再** 按 push 自动创建整线 `main -> sb4` 或 `sb4 -> sb3` 同步 PR
- 如需发起 `sb3` 维护回补 PR，应先准备好基于 `sb3` 的 backport 分支，再使用手工 workflow 或 `gh pr create`

设计原则：

- 回补必须显式、选择性、可追踪
- 自动化只能辅助“已准备好的 backport 分支开 PR”，不能替代人工挑选回补范围
- `sb3` 与 `sb4` 的验证都必须在各自目标分支语义下重新执行

## 9. 当前稳态结论

- `sb4`：业务主干
- `sb3`：维护线
- `main`：历史入口，不作为默认开发目标
- 回补方式：`sb4` 收敛后，按需 `cherry-pick` / backport 到 `sb3`
- 禁止动作：把整条 `sb4` 周期性同步到 `sb3`

# Permission Refactor Signal Source Rules

## Purpose

Avoid mixed signal interpretation by splitting all permission-refactor signals into:

- `TEST`: generated by smoke/E2E/mutate-restore/assertion traffic
- `RUNTIME`: generated by real gray runtime traffic

## Fixed Enum

`signal_source` only accepts:

- `TEST`
- `RUNTIME`

No empty value, no third enum.

## Transport Rules

Primary mechanism (A):

- Request header: `X-Permission-Signal-Source`
  - test traffic must set: `TEST`
  - runtime traffic defaults to `RUNTIME` when header absent

Fallback mechanism:

- JVM property: `-Dpermission.signal.source=TEST|RUNTIME`
- Env var: `PERMISSION_SIGNAL_SOURCE=TEST|RUNTIME`

## Logging Requirements

The following logs must contain `signalSource`:

- `OLD_PERMISSION_ONLY` (旧口径: `OLD_FALLBACK`)
- `DENY_DISABLED`
- `DENY_UNKNOWN`
- `permissionsVersion inputs summary`

## Script Requirements

Smoke/E2E runners must force TEST source:

- `run-permission-dev-smoke-10m.sh`
- `scripts/e2e/run-permission-refactor-e2e.sh`

with Maven option:

- `-Dpermission.signal.source=TEST`

## Governance Decision Rule

Gray expansion decisions are based primarily on `RUNTIME` signals.

`TEST` signals remain valid for regression quality checks, but must not be used as the sole block for expansion decisions.

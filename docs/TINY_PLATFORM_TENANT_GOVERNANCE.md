# Tiny Platform Tenant Governance

> 目标：明确多租户环境下租户生命周期、治理操作、权限与审计约束，为后续实现与评审提供统一基线。  
> 关联文档：`TINY_PLATFORM_TENANT_NAMING_GUIDELINES.md`、`TINY_PLATFORM_AUTHORIZATION_MODEL.md` §5.2、`saas-multi-tenant-issues-analysis.md`。

---

## 1. 租户生命周期阶段（目标态）

| 阶段 | 描述 | 设计要点（借鉴多租户最佳实践） |
|------|------|------------------------------|
| PENDING_CREATE / TRIAL | 请求已接收但尚未完成 bootstrap；或试用期租户 | 需考虑自助/人工开通流程、试用配额与有效期、数据驻留与合规要求；在该阶段应准备好模板数据库、预设 plan/配额、试用限制和到期策略。 |
| ACTIVE | 正常运行状态，允许登录、业务写入、调度/导出等能力 | 需要持续的容量规划与“噪声邻居”治理；应在文档/运维手册中说明监控指标、告警策略、版本升级与变更窗口管理。 |
| FROZEN / DEACTIVATED | 冻结/停用状态，不再允许新登录和写操作，保留只读访问 | 适用于客户暂停订阅、风险排查或欠费缓冲期；应定义最长停留时间和自动转 DECOMMISSIONED 的条件，冻结期间计费策略可按产品决定（降费或暂停）。 |
| DECOMMISSIONED / OFFBOARD | 已下线状态，仅保留合规期内所需的审计/账单/归档数据，不再对外提供业务访问 | 应在进入该状态前完成账单结算和必要的数据导出/迁移；需定义数据保留期、再入驻策略以及归档位置，避免无界期的历史租户残留。 |

> 说明：当前实现仅显式区分 `enabled` 与否；上述更细颗粒度状态用于指导后续 schema/实现演进，并对齐业界对租户生命周期（入驻、运行、暂停、下线）的最佳实践。

## 2. 治理操作矩阵（草案）

| 操作 | 允许发起者 | 影响范围 | 基本约束 | 审计要点 |
|------|------------|----------|----------|----------|
| 创建租户 | 平台租户内具备租户管理权限的管理员（`@tenantManagementAccessGuard`） | 新建 `tenant` 记录、触发 `TenantBootstrapServiceImpl` 从平台模板派生角色/资源/菜单 | 禁止在普通业务租户内创建其他租户；必须通过平台租户控制面执行；需幂等保护（已有 `@Idempotent`） | 记录 `createdTenantId`、发起人、plan、配额、来源 IP 等 |
| 更新租户（名称/域名/配额/plan） | 同上 | 更新 `tenant` 基本信息及配额字段 | 不得在业务 API 中直接修改租户配额，仅允许通过控制面；变更配额需同步更新相关限流/调度/导出策略（后续实现） | 记录变更前后 diff、发起人、时间点 |
| 冻结租户（ACTIVE → FROZEN） | 平台管理员 | 限制该租户的登录、业务写入、调度/导出执行 | - 登录入口应拒绝该租户的新会话创建；<br>- 业务写 API 必须 fail-closed；<br>- 调度/导出 job 可选暂停或仅允许查看历史 | 记录冻结原因、发起人、影响模块、预计恢复时间（如有） |
| 解冻租户（FROZEN → ACTIVE） | 平台管理员 | 恢复登录与业务写入能力 | 仅在风险排查完成后执行；如涉及账单/合规，需要额外确认 | 记录解冻原因、发起人、时间点 |
| 标记下线（DECOMMISSIONED） | 平台管理员 + 合规流程 | 停止对外服务，仅保留审计/账单/归档访问 | 应先冻结，再完成数据导出/迁移，最后标记下线；不建议直接物理删除业务数据 | 记录决策依据、保留期限、归档位置 |

## 3. 与平台租户语义的关系

当前迁移期平台语义由 `PlatformTenantProperties.platformTenantCode` 提供（默认 `"default"`），并通过：

- `TenantManagementAccessGuard`：仅允许“平台租户 + ROLE_ADMIN”访问 `/sys/tenants` 控制面；
- `TenantBootstrapServiceImpl`：优先使用 `tenant_id IS NULL` 的平台模板资源/角色，若不存在则回退到配置的平台租户 code。

长期目标（见 `TINY_PLATFORM_AUTHORIZATION_MODEL.md` §5.2）：

- 平台语义由 `scope_type=PLATFORM` 表达，而不是特定租户 code；  
- 平台租户仅作为控制面的一个逻辑视图，不再承载全部平台模板数据。

## 4. 后续实施建议（非本轮必做）

1. 在 `tenant` 表中显式引入状态字段（如 `lifecycle_status`：PENDING_CREATE/ACTIVE/FROZEN/DECOMMISSIONED），并在 `TenantService` 层统一维护，每个状态设定最大停留时间与转换条件（例如 FROZEN 最长 N 天，逾期自动转 DECOMMISSIONED 并启动保留期计时）。  
2. 在认证链路与关键业务入口（调度、导出等）增加对 FROZEN/DECOMMISSIONED 状态的 fail-closed 校验：  
   - ACTIVE：允许正常读写；  
   - FROZEN：阻止新会话和写操作，仅允许只读访问；  
   - DECOMMISSIONED：仅允许平台管理员或合规角色查看/导出必要信息，拒绝业务访问。  
3. 为冻结/下线操作补充集成测试或 real-link E2E，覆盖：  
   - 冻结后登录被拒绝；  
   - 冻结后写入被拒绝、只读访问仍可用；  
   - 标记下线后，控制面不可再对该租户执行非只读操作。  
4. 配合计费系统：冻结状态应停止计费或切换到最低收费档位；解冻时恢复计费；下线前需结清账单并提供账单导出路径，与租户生命周期事件打通。  
5. 针对扩容、迁移、地域搬迁、合并与拆分等高级运维操作，在后续版本补充专门章节，定义：  
   - 扩容/迁移时的停机窗口与数据一致性保障；  
   - 跨地域迁移的数据驻留与路由切换策略；  
   - 租户合并/拆分时的用户/数据/权限合并策略以及命名冲突解决办法。  
6. 将本文件的矩阵与现有规则文档（90/91/93）对齐，必要时在规则中增加对应 Must/Should 约束，并在“待定决策”中列出：冻结期限、再激活流程、高风险操作审批链（冻结/下线/地域搬迁/合并拆分）等需要在后续设计中细化的策略。

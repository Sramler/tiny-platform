# 调度管理任务示例设计

本文档提供若干**具有实际业务含义**的任务示例，用于充分展示当前调度管理的：任务类型、任务、DAG 编排、依赖执行、手动/定时触发、运行历史、节点级操作（触发/重试/暂停/恢复）等能力。

---

## 一、可用执行器与参数

| 执行器 identifier     | 说明           | 常用参数示例 |
|----------------------|----------------|--------------|
| `loggingTaskExecutor` | 将参数打日志并返回 | `message`, `step`, 任意键值 |
| `delayTaskExecutor`   | 延迟 N 毫秒后返回，可模拟失败 | `delayMs`（必填）, `fail`（true 模拟失败） |
| `shellExecutor`       | Shell 占位（当前仅打日志） | `command`, `args` 等 |

---

## 二、示例场景总览

| 场景       | 业务含义           | 展示能力                         | 节点数 |
|------------|--------------------|----------------------------------|--------|
| 数据导出流水线 | 准备→校验→执行     | 串行依赖、多节点、日志步骤       | 3      |
| 报表计算流水线 | 并行报表→汇总      | 多前驱、并行、Delay 模拟耗时     | 3      |
| 健康检查       | 检测→记录          | Cron 定时、简单链、手动/定时触发 | 2      |
| 最简示例（现有） | 准备→执行          | 最简 DAG、运行历史、节点操作     | 2      |

## 三、示例一：数据导出流水线

**业务含义**：模拟「数据导出」流程：先准备导出环境，再校验数据，最后执行导出。三步串行，每步用日志记录，便于在运行历史与日志中观察顺序。

### 3.1 任务类型（1 个）

| 字段 | 值 |
|------|-----|
| 编码 | `EXPORT_PIPELINE` |
| 名称 | 数据导出流水线-步骤 |
| 描述 | 用于数据导出流程中的每一步，打日志并返回，便于追踪 |
| 执行器 | `loggingTaskExecutor` |
| 默认超时(秒) | 120 |
| 默认最大重试 | 1 |

### 3.2 任务（3 个）

| 编码 | 名称 | 描述 | 参数(JSON) |
|------|------|------|------------|
| `export_prepare` | 导出-准备 | 准备导出目录与权限 | `{"message":"准备导出环境","step":"prepare"}` |
| `export_validate` | 导出-校验 | 校验数据范围与一致性 | `{"message":"校验导出数据","step":"validate"}` |
| `export_execute` | 导出-执行 | 执行导出并落盘 | `{"message":"执行导出写入","step":"execute"}` |

### 3.3 DAG 编排

- **DAG 名称**：数据导出流水线  
- **DAG 编码**：`export_pipeline`  
- **节点与边**：  
  - 节点 `prepare` → 任务「导出-准备」  
  - 节点 `validate` → 任务「导出-校验」  
  - 节点 `execute` → 任务「导出-执行」  
  - 边：`prepare → validate`，`validate → execute`  

**可展示**：运行一次后，在运行历史中看到 3 个节点按顺序执行；日志中可见三步的 `message` 与 `step`。

## 四、示例二：报表计算流水线

**业务含义**：模拟「日报/周报」：用户统计与订单统计**并行**计算，两者都完成后再执行「汇总」步骤（打汇总日志）。用 Delay 执行器模拟统计耗时。

### 4.1 任务类型（2 个）

**类型 A：报表统计（Delay）**

| 字段 | 值 |
|------|-----|
| 编码 | `REPORT_STAT` |
| 名称 | 报表统计-模拟计算 |
| 执行器 | `delayTaskExecutor` |
| 默认超时(秒) | 300 |
| 默认最大重试 | 1 |

**类型 B：报表汇总（Logging）**

| 字段 | 值 |
|------|-----|
| 编码 | `REPORT_SUMMARY` |
| 名称 | 报表汇总-写日志 |
| 执行器 | `loggingTaskExecutor` |
| 默认超时(秒) | 60 |

### 4.2 任务（3 个）

| 编码 | 名称 | 任务类型 | 参数(JSON) |
|------|------|----------|------------|
| `report_user` | 报表-用户统计 | 报表统计-模拟计算 | `{"delayMs":2000,"message":"用户统计"}` |
| `report_order` | 报表-订单统计 | 报表统计-模拟计算 | `{"delayMs":1500,"message":"订单统计"}` |
| `report_summary` | 报表-汇总 | 报表汇总-写日志 | `{"message":"日报汇总完成","step":"summary"}` |

### 4.3 DAG 编排

- **DAG 名称**：报表计算流水线  
- **DAG 编码**：`report_pipeline`  
- **节点**：  
  - `user_report` → 任务「报表-用户统计」  
  - `order_report` → 任务「报表-订单统计」  
  - `summary` → 任务「报表-汇总」  
- **边**：`user_report → summary`，`order_report → summary`  

**可展示**：`user_report` 与 `order_report` 并行执行，两者都成功后 `summary` 才执行；运行历史中可见并行与汇总顺序；Delay 可在日志中看到耗时。

## 五、示例三：健康检查

**业务含义**：定时对系统做健康检查（模拟），然后记录结果。用于展示 **Cron 定时触发** 与简单两节点链。

### 5.1 任务类型（1 个）

| 字段 | 值 |
|------|-----|
| 编码 | `HEALTH_CHECK` |
| 名称 | 健康检查-记录 |
| 执行器 | `loggingTaskExecutor` |
| 默认超时(秒) | 30 |

### 5.2 任务（2 个）

| 编码 | 名称 | 参数(JSON) |
|------|------|------------|
| `health_ping` | 健康检查-检测 | `{"message":"Ping 检测","step":"ping"}` |
| `health_log` | 健康检查-记录 | `{"message":"记录健康结果","step":"log"}` |

### 5.3 DAG 编排

- **DAG 名称**：健康检查  
- **DAG 编码**：`health_check`  
- **节点**：`ping` → 任务「健康检查-检测」，`log` → 任务「健康检查-记录」  
- **边**：`ping → log`  
- **Cron（可选）**：`0 */5 * * * ?` 表示每 5 分钟执行一次（仅当 DAG 保存时配置了 cron 且 Quartz 启用时生效）

**可展示**：手动触发一次可见两节点顺序执行；若配置 Cron，可观察定时触发的 Run 与运行历史。

## 八、如何创建这些示例

### 方式一：在前端按文档创建（推荐理解流程）

1. **任务类型**：在「任务类型」页按上表创建 1～2 个任务类型（执行器选 `loggingTaskExecutor` / `delayTaskExecutor`）。  
2. **任务**：在「任务管理」页选择对应任务类型，创建上述任务并填写参数 JSON。  
3. **DAG**：在「DAG 管理」新建 DAG，进入 DAG 详情 → 版本 → 添加节点（绑定任务）、添加边；可选填 Cron。  
4. **触发**：在 DAG 列表或详情中「手动触发」，再到「运行历史」查看 Run 与节点执行情况。

### 方式二：执行 SQL 导入（示例一 + 示例二 + 示例三）

**推荐：清理旧数据并插入新示例（可重复执行）**

```bash
# 库名与 data.sql 一致（如 tiny_web）
mysql -u user -p tiny_web < tiny-oauth-server/src/main/resources/db/changelog/scheduling-reset-and-examples.sql
```

脚本路径：**`tiny-oauth-server/src/main/resources/db/changelog/scheduling-reset-and-examples.sql`**。  
脚本会：先删除所有 `scheduling_*` 表数据及旧 DAG 的 Quartz Job（`dag-1`），再插入上述三个示例（任务类型 4 条、任务 8 条、DAG 3 个、版本/节点/边若干）。依赖 `tenant_id=1` 已存在。

**仅追加插入（不删旧数据）**：可使用 `docs/scheduling-task-examples-data.sql`（INSERT，与现有 demo id=1 不冲突）。
